About This Project

Purpose

This proof of concept demonstrates a production-grade 3-tier architecture on AWS, showcasing the integration of compute, caching, and database services with proper network isolation, security, and observability.

The architecture follows AWS Well-Architected Framework principles and implements industry best practices for security, performance, and cost optimization.

Caching Strategy

This application implements the cache-aside (lazy loading) pattern, which is the most common caching strategy for read-heavy workloads:

  • Read Path: Check cache first → if miss, query database → store in cache → return
  • Write Path: Write to database → invalidate cache
  • Benefits: Only requested data is cached, cache failures don't break the app
# Cache-aside pattern example cached = await redis.get(key) if cached: return json.loads(cached) # Cache hit data = await db.fetch(query) # Cache miss - query DB await redis.setex(key, TTL, json.dumps(data)) # Store in cache return data

Network Architecture

Subnet Type Resources Internet Access
Public ALB, NAT Gateway Yes (inbound + outbound)
Private EC2 (Application) Outbound only (via NAT)
Isolated RDS, ElastiCache None

Security Layers

  • WAF: AWS Managed Rules (CRS, SQLi, Known Bad Inputs) + Rate limiting
  • CloudFront: DDoS protection, SSL/TLS termination
  • Security Groups: ALB → EC2 → RDS/Redis (strict ingress rules)
  • Secrets Manager: Database credentials with automatic rotation capability
  • Encryption: At rest (RDS, S3, EBS) and in transit (TLS)
  • IAM: Least privilege roles for EC2

Note: NAT Gateway is the largest cost. For production, consider NAT instances or AWS PrivateLink to reduce costs.

Production Considerations

  • Multi-AZ: Enable RDS Multi-AZ and ElastiCache replication
  • Auto Scaling: Add ASG for EC2 instances
  • Monitoring: CloudWatch dashboards, alarms, and X-Ray tracing
  • Backup: Configure RDS automated backups and Redis snapshots
  • DR: Cross-region replication for critical data
